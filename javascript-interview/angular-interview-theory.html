Lifecycle Hooks +
               Common hooks:

                ngOnInit – Runs once after the component is initialized; used for initial data loading.
                           Call API to load data when component loads

                ngOnChanges – Runs whenever an input property changes; used to react to input updates.
                              Update UI when parent passes new data

                ngAfterViewInit – Runs after the component’s view and child views are fully initialized.
                                  Access ViewChild or manipulate DOM

                ngOnDestroy – Runs just before the component is destroyed; used for cleanup like unsubscribing.
                            Unsubscribe from observables or clear timers

            | Hook                                  | When It Runs                                                                | Use Case                                              |
            | ------------------------------------- | --------------------------------------------------------------------------- | ----------------------------------------------------- |
            | `ngOnChanges(changes: SimpleChanges)` | Called **when an input property changes** (before `ngOnInit`)               | React to `@Input()` changes                           |
            | `ngOnInit()`                          | Called **once after first `ngOnChanges`**                                   | Initialize component, fetch data                      |
            | `ngDoCheck()`                         | Called **during every change detection cycle**                              | Detect/act on changes manually                        |
            | `ngAfterContentInit()`                | Called **after content (ng-content) is projected**                          | Access projected content                              |
            | `ngAfterContentChecked()`             | Called **after every check of projected content**                           | Custom content change detection                       |
            | `ngAfterViewInit()`                   | Called **after component’s view (template + child components) initialized** | Access `@ViewChild`/`@ViewChildren` safely            |
            | `ngAfterViewChecked()`                | Called **after every check of the component’s view**                        | Detect changes in child components                    |
            | `ngOnDestroy()`                       | Called **just before component is destroyed**                               | Cleanup subscriptions, timers, detach event listeners |

ViewEncapsulation +
            Controls how a component’s CSS styles are applied and scoped.

            Emulated (default) – Styles are scoped to the component by adding unique attributes (prevents style leakage).

            None – Styles are global and apply to the entire application.

            ShadowDom – Uses the browser’s native Shadow DOM to strictly isolate component styles.



            When to Use

            Emulated – Most cases (safe and default)

            None – Shared/global styling (themes, layouts)

            ShadowDom – Web components or strict style isolation


            | Type                   | CSS Scope      | Style Leakage | Uses Shadow DOM | Use Case             |
            | ---------------------- | -------------- | ------------- | --------------- | -------------------- |
            | **Emulated** (default) | Component only | ❌ No          | ❌ No            | Normal Angular apps  |
            | **None**               | Global         | ✅ Yes         | ❌ No            | Global themes/layout |
            | **ShadowDom**          | Component only | ❌ No          | ✅ Yes           | Web components       |

ChangeDetectionStrategy.OnPush +

ChangeDetectionStrategy.OnPush OnPush tells Angular to run change detection only when necessary, improving performance. 

            Runs change detection only when:

            -> Input reference changes – A new object/array reference is passed from the parent

            -> Event occurs – A user action like click or input inside the component

            -> Observable emits value – Data updates via async pipe or manual subscription

Custom Directive +
            A custom directive is a user-defined Angular directive used to add custom behavior or manipulate the DOM of an element.

            Custom directives let us create reusable DOM behavior in Angular.
   
            Used to manipulate DOM behavior (e.g., highlight, tooltip).


            Example:

            TS File Code:
            
            import { Directive, ElementRef, HostListener } from '@angular/core';

            @Directive({
              selector: '[appHighlight]'
            })
            export class HighlightDirective {

              constructor(private el: ElementRef) {}

              @HostListener('mouseenter')
              onMouseEnter() {
                this.el.nativeElement.style.backgroundColor = 'yellow';
              }

              @HostListener('mouseleave')
              onMouseLeave() {
                this.el.nativeElement.style.backgroundColor = '';
              }
            }

Dependency Injection +
          Dependency Injection (DI) is a design pattern where a class does not create its dependencies itself, but receives them from an external source.


           Design pattern where dependencies are injected instead of created manually.


            Benefits of Dependency Injection
            | Benefit        | Explanation                                                |
            | -------------- | ---------------------------------------------------------- |
            | Loose coupling | Components don’t create dependencies themselves            |
            | Easy testing   | Can replace dependencies with mocks/fakes                  |
            | Reusability    | Same service instance can be shared                        |
            | Configurable   | Can have different instances in different parts of the app |

AOT in Angular+
           AOT (Ahead-of-Time compilation) is an Angular process where Angular compiles HTML templates and components into efficient JavaScript code during the build time instead of at runtime in the browser.


          | Feature          | AOT              | JIT               |
          | ---------------- | ---------------- | ----------------- |
          | Compilation time | Build time       | Runtime (browser) |
          | App startup      | **Faster**       | Slower            |
          | Bundle size      | **Smaller**      | Larger            |
          | Error detection  | **During build** | During runtime    |
          | Security         | **More secure**  | Less secure       |
          | Production use   | ✅ Default        | ❌ Not recommended |



          Interview-Ready Definition

          AOT (Ahead-of-Time compilation) compiles Angular templates into optimized JavaScript at build time, resulting in faster startup, smaller bundles, improved security, and early error detection compared to JIT compilation.
IVY in Angular+
          Ivy is Angular’s default rendering and compilation engine that converts templates into efficient JavaScript instructions, enabling faster builds, smaller bundles, improved performance, and better debugging. It works together with AOT to optimize Angular applications.
Auth Guard in Angular+
              Run Command ng generate guard auth

              create Services

              // auth.service.ts
              import { Injectable } from '@angular/core';

              @Injectable({
                providedIn: 'root'
              })
              export class AuthService {

                isLoggedIn(): boolean {
                  return !!localStorage.getItem('token'); // example
                }
              }

              Implement Authguard

              // auth.guard.ts
            import { inject } from '@angular/core';
            import { CanActivateFn, Router } from '@angular/router';
            import { AuthService } from './auth.service';

            export const authGuard: CanActivateFn = () => {
              const authService = inject(AuthService);
              const router = inject(Router);

              if (authService.isLoggedIn()) {
                return true;
              }

              router.navigate(['/login']);
              return false;
            };


            // app-routing.module.ts
            import { NgModule } from '@angular/core';
            import { RouterModule, Routes } from '@angular/router';
            import { AuthGuard } from './auth.guard';
            import { DashboardComponent } from './dashboard/dashboard.component';

            const routes: Routes = [
              {
                path: 'dashboard',
                component: DashboardComponent,
                canActivate: [AuthGuard]
              },
              {
                path: 'login',
                loadComponent: () => import('./login/login.component')
              }
            ];

            @NgModule({
              imports: [RouterModule.forRoot(routes)],
              exports: [RouterModule]
            })
            export class AppRoutingModule {}
CanActivate, CanActivateChild, canDeactivate, CanLoad+
             CanActivate
            👉 Controls access to a route
            Use case:
            ✔ User must be logged in
            ✔ Role-based access


            CanActivateChild
            👉 Controls access to child routes
            Use case:
            ✔ Protect all child routes with one guard
            ✔ Admin section protection


            CanDeactivate
            👉 Prevents leaving a route
            Use case:
            ✔ Unsaved form changes
            ✔ Confirmation before exit


            CanLoad (⚠️ Legacy / Lazy Modules)
            👉 Prevents loading a lazy-loaded module
            Use case:
            ✔ Block downloading admin module for unauthorized users
